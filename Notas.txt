Tempo de vida útil dos Serviços

Quando registamos serviços em um container, precisamos definir o tempo de vida que queremos usar para este serviço.
O tempo de vida do serviço controla por quanto tempo um objeto vai existir após ter sido criado pelo container
O tempo de vida pode ser definido usando o metodo de extensão apropriado no IServiceCollection ao registar o serviço.

Transient: São criados cada vez que são solicitados. Cada vez que se injeta o serviço em uma classe, será criada uma nova instância do serviço. É indicado
para serviços leves e sem estado.
São registados usando o metodo AddTransient

Scoped: São criados em cada solicitação (uma vez por solicitação do cliente). É indicado para aplicações WEB.
Se durante um request se usar a mesma injecção de dependência em muitos lugares, vai-se usar a mesma instância dos objetos, 
e ele fará referencia á mesma alocação de memória.  
São registados usando o método AddScoped

Singleton: São criados uma vez durante a vida útil da aplicação que usa a mesma instância para toda a aplicação.
São registados usando o método AddSingleton

A informação é então passada de volta para a camada de negócios para processamento e então apresentada ao utilizador.

Atributos:  Key - Identifica a propriedade como uma chave primaria na tabela
		    Table("nome") - Define o nome da tabela para a qual a classe será mapeada

			DataType - Associa um tipo de dados adicional a uma propriedade

Model Binding

O model binding ou vinculação de modelos é um recurso que nos permite mapear dados de uma requisição HTTP 
para parâmeros de uma Action de um Controlador.
Esse mapeamento inclui todos os tipos de parâmetros: números, strings, arrays, listas, tipos complexos, 
listas de objetos, etc.

Como Funciona?: Quando nossa API recebe uma requisição HTTP, ela a roteia para um método Action específico de um controlador com base na rota definida.
Ela determina qual o método Action será executado com base no que está definido na rota e , em seguida, vincula os valores da requisição HTTP aos parâmetros desse método Action.

1 - Valores formulário: (POST e PUT) - Enviados no corpo do request
2 - Rotas: [Route("api/[Controller]")] ou HttpGet("{id}")
3 - Query Strings: api/produtos/4?nome=Sumo&ativo=true

Unit Of Work

- Gerir as transações
- Ordenar o CRUD na base dados
- Impedir a concorrência(duplicação de atualizações)
- Usar somente uma instância do contexto por requisição

Os repositórios são responsáveis por realizar operações de leitura e escrita na base dados, mas não devem chamar o SaveChanges.
Eles simplesmente interagem com o DbContext, modificam objetos e preparam as mudanças para persistência.
A Unit of Work é responsável por gerir transações e, mais importane, por chamar o SaveChanges.
Ela é quem inicia, confirma ou reverte a transação e garante que todas as operações realizadas nos 
repositórios sejam incluídas em uma única transação da base de dados.

Lazy Loading: significa adiar a obtenção dos objetos até que sejam relamente necessários.

Autenticação e Autorização

1 - Anónimo: Uma requisição anínima não contem informações de autenticação
2 - Basic - A autenticação básica envia uma cadeia de carateres codificada em Base64 que contem um nome de utilizador e senha para o cliente
3 - Tokens(Bearer) - É um esquema de autenticação HTTP que envolve tokens de segurança chamados tokens de portador.(bearer token)

Bearer Token: é uma string codificada, geralmente gerada pelo servidor em resposta a uma requisição de login.

JWT - JSON Web Token
-Padrão que define uma forma segura de transmitir informações entre duas partes.
-A trasmissão pode ser feita via URL, POST, ou em cabeçalho HTTP e a informação é assinada digitalmente por um algoritmo
ou par de chaves public/privada usando RSA(sistema de criptografia de chave publica)
-São usado assim tokens criptografados para permitir o acesso aos recursos de uma Web Api - (abordagem conhecida como Bearer Authentication)

Token JWT
Header - Contém o tipo de token(JWT) e a criptografia usada(HMAC ou RSA)
Payload - Contém os pedidos. São declarações ou claims associadas ao utilizador do token
Signature - Assinatura usada na validação do token usando uma chave secreta

Para gerar a assinatura usamos o Header e o Pauload codificando-os e a seguir assinamos usando o algoritmo definido.

Etapas do proesso da implementação da autenticação via token JWT

1. Geração do Token(Refresh Token)
- verificar credenciais do utilizador(login)
- definir as claims do utilizador
- definir a chave secreta e o algoritmo de encriptação usado(HmacSha256)
- Gerar o token com base no emissor, audiência, claims, chave secreta e feinir data de expiração do token

2. Validação do token(middleware)
- validar emissor(quem emite o token)
- validar audiência (quem pode usar o token)
- validar assinatura com chave secreta
- validar tempo de vida(tempo de expiração do token)

Consiste em retringir o número de requisições que podem ser feitas em um período de tempo a uma aplicação
Motivação: 	- Evitar a sobrecarga de servidores ou aplicativos

VERSIONAMENTO DE SOFTWARE

Os números de versão geralmente seguem um padrão como o parão Major.Minor.Patch

1. Major: Aumentado quando são feitas alterações sgnificativas, como adição de grandes funcionalidades, mudanças arquiteturais
ou alterações incompatíveis com versões anteriores.

2. Minor: Aumentado para adições de funcionalidades menores ou melhorias, geralmente compatíveis com versões anteriores.

3. Patch: Aumentado para correções de bugs ou pequenas melhorias que não afetam a compatibilidade com versões anteriores.

CACHE(MemoryCache)
O cache é um mecanismo que armazena cópias de dados frequentemente acessados 
em um local de acesso mais rápido do que a fonte original.
Uma técnica usada para armazenar temporariamente dados ou respostas acessadas com frequência, com o objetivo de melhorar o desempenho
e reduzir a carga nos recursos computacionais, como a base dados ou serviçoes externos.

Quando usar?:
- Para dados frequentemente acessados e raramente alterados
- Para listas de configurações ou opções
- Melhorar o desempenho reduzindo consultas repetitivas á base de dados

Quando Não usar?: 
- Quando os dados forem sensiveis ou criticos e precisam sempre ser atualizados
- Quando a aplicação depende de respostas baseadas em tempo real
- Quando o consumo de memória do servidor for um fator critico
